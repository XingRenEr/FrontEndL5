# HTTP/HTTPS 完整详解（加密体系+握手流程+安全机制）
## 一、HTTPS 三大基础加密算法
HTTPS = HTTP + TLS/SSL，底层靠**对称加密、非对称加密、Hash哈希**三类算法组合实现安全，三者各司其职。

### 1. 对称加密（会话数据加密）
#### 核心定义
加密、解密使用**同一套密钥**，加解密速度极快，适合传输网页大量明文数据（请求参数、页面内容、Cookie）。
#### 优缺点
- 优点：运算效率高，海量数据传输首选
- 缺点：密钥分发难题，客户端和服务端如何安全同步密钥？直接网络传输密钥会被窃听，单独无法使用。
#### 常见算法
AES（主流）、DES、3DES
#### HTTPS作用
TLS握手完成后，**后续所有业务数据全部用对称加密传输**。

### 2. 非对称加密（密钥协商、身份认证）
#### 核心定义
一对密钥：**公钥（公开分发）、私钥（服务器独有，绝不外传）**
- 公钥加密的数据，只能用对应私钥解密
- 私钥加密的数据，只能用对应公钥解密
#### 优缺点
- 优点：解决密钥分发安全问题，可做身份校验
- 缺点：运算速度极慢，不适合加密大体积网页数据
#### 常见算法
RSA、ECC（椭圆曲线，移动端轻量化）
#### HTTPS作用
1. 客户端用服务器公钥加密「随机预主密钥」，传给服务端，只有服务端私钥能解密；
2. 配合数字证书校验服务器身份。

### 3. Hash哈希算法（完整性校验+数字签名）
#### 核心定义
单向不可逆摘要算法：任意长度输入，输出固定长度哈希值；原文改动1字符，哈希值完全变化，**无法反向还原原文**。
通常搭配加盐、HMAC使用。
#### 优缺点
- 优点：快速校验数据是否被篡改；用于签名校验
- 缺点：单纯哈希无法防伪造，需配合私钥签名
#### 常见算法
SHA256（主流）、MD5（已淘汰不安全）
#### HTTPS作用
1. 校验传输报文完整性，中间人篡改数据后哈希不匹配，连接直接中断；
2. 作为数字签名的摘要载体，CA证书签名、握手报文签名都依赖哈希。

## 二、HTTPS 安全核心机制
### 4. 混合加密（HTTPS核心方案）
单独对称/非对称加密都有短板，HTTPS采用**两者结合的混合加密**，兼顾安全与性能：
1. 握手阶段：用**非对称加密**安全交换「对称会话密钥」；
2. 数据传输阶段：用**对称加密**加密HTTP业务数据；
3. 全程搭配Hash算法校验数据完整性。

流程简化：
客户端生成随机密钥 → 服务器公钥加密随机密钥传输 → 服务器私钥解密得到随机密钥 → 双方用该随机密钥（对称密钥）加密通信。

### 5. 数字签名（防篡改、防伪造）
解决两个问题：数据是否被篡改、发送方身份是否真实。
#### 签名完整流程（服务器/CA签发证书通用逻辑）
1. 原文 → Hash生成摘要；
2. 摘要用**私钥**加密，生成数字签名；
3. 原文+签名一起发送。
#### 校验流程
接收方：
1. 原文重新Hash生成新摘要；
2. 使用对方公钥解密签名，得到原始摘要；
3. 两份摘要对比，一致=未篡改、身份可信。

HTTPS场景：CA机构用自身私钥给网站证书签名，浏览器用CA内置公钥验证证书合法性。

### 6. CA机构（信任根，解决身份信任问题）
#### 问题背景
如果仅靠公钥加密，中间人可以劫持流量，替换服务器公钥（中间人攻击），客户端无法分辨公钥真假。
#### CA作用
第三方权威可信机构，负责颁发、签名、管理数字证书，建立信任链：
1. 网站向CA提交域名、企业信息，证明域名所有权；
2. CA审核通过，用自身**根私钥**签发网站数字证书；
3. 证书包含：网站公钥、域名、有效期、CA数字签名；
4. 操作系统/浏览器内置全球可信CA根证书（内置CA公钥）；
5. 客户端连接网站时，下载网站证书，用内置CA公钥校验证书签名，确认公钥属于目标网站，不是中间人伪造。

#### 分类
根CA、中级CA、域名证书（DV）、企业证书（OV）、增强证书（EV）

### 7. 重放攻击 & HTTPS防御方案
#### 什么是重放攻击
中间人窃听完整握手/请求报文，后续重复发送该报文，欺骗服务器重复执行业务（重复下单、重复登录）。
#### HTTPS防御手段
1. TLS握手随机数：客户端随机数+服务端随机数，每次握手唯一，旧报文无法复用；
2. 会话序列号：每条TLS报文自带递增序号，重复报文直接丢弃；
3. 时间戳、一次性临时密钥；
4. 业务层搭配请求Nonce、时效Token双重防护。

## 三、HTTPS RSA四次握手完整流程（TLS1.2经典RSA握手）
> TLS1.3简化为2次握手，面试常考TLS1.2 RSA四次握手，分4步，全程基于混合加密、数字证书、RSA非对称算法。
> 前置：服务器已部署CA颁发的数字证书（含服务器公钥）

### 第一次握手：客户端 → 服务端 Client Hello
客户端发送：
1. 支持的TLS版本（TLS1.0/1.1/1.2）；
2. 客户端随机数 Client Random（32字节，后续生成会话密钥）；
3. 客户端支持的加密套件列表（AES-RSA、ECC等）；
4. 扩展字段（SNI域名，一台服务器多证书场景）。

### 第二次握手：服务端 → 客户端 Server Hello + 证书下发
服务端返回多条消息合并：
1. Server Hello：选定双方兼容的最高TLS版本、选定加密套件、**服务端随机数 Server Random**；
2. 数字证书：下发网站CA证书（内含服务器公钥+CA签名）；
3. Server Hello Done：告知客户端握手信息发送完毕。

客户端此时操作：
1. 使用浏览器内置CA根公钥，校验证书CA签名，确认证书合法、域名匹配；
2. 提取证书内**服务器公钥**。

### 第三次握手：客户端 → 服务端 Pre-Master Secret（核心密钥交换）
1. 客户端生成48字节随机预主密钥 `Pre-Master Secret`；
2. 使用**服务器公钥（RSA非对称加密）**加密预主密钥，发送给服务端；
3. 附带报文：客户端密钥交换完成通知。

> 关键：只有服务器独有私钥，能解密拿到预主密钥，中间人截获也无法破解。

### 第四次握手：服务端解密密钥，双方生成会话密钥，通知加密通信
1. 服务器用自身RSA私钥解密，拿到预主密钥；
2. 客户端、服务器双方使用「Client Random + Server Random + Pre-Master Secret」，通过伪随机函数推导**统一对称会话密钥**；
3. 服务端发送 `Change Cipher Spec`：告知客户端后续全部使用对称加密传输；
4. 发送 `Finished` 加密报文（Hash校验值），客户端校验握手报文是否完整未篡改；
5. 客户端同样回复 `Change Cipher Spec + Finished`；

#### 握手结束标志
两端 `Finished` 校验通过，TLS握手完成，后续HTTP请求全部通过**AES对称加密**传输，HTTPS安全通道建立成功。

## 补充拓展
1. RSA握手短板：RSA不支持前向保密，私钥泄露后历史流量可被解密；现代网站改用ECDHE握手（临时椭圆曲线密钥）；
2. 证书失效风险：证书过期、域名不匹配、CA不信任都会触发浏览器安全告警；
3. 混合加密逻辑总结：非对称只用来交换对称密钥，对称负责真实业务传输，Hash全程校验防篡改，CA保证公钥可信。
